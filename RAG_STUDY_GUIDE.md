# RAG 같이 공부하기

> 외부 자료보다, **우리가 만든 `app.py`를 텍스트 삼아** 직접 뜯어보는 게 제일 빠름. 41청크로 실제 돌아가는 코드가 이미 있음.

## 1. 개념 (10분, 발표 담당자가 말로 설명)

**RAG = Retrieval-Augmented Generation.** LLM이 답할 때 머릿속 지식만 쓰지 않고, **외부 문서에서 관련 조각을 검색해 프롬프트에 붙여넣고** 답하게 하는 구조.

```
문서 → 청킹 → 임베딩 → 저장 → (질문) → 질문도 임베딩 → 유사도 검색(top-k) → 프롬프트에 근거 주입 → LLM 생성
```

왜 필요하냐: LLM이 지어내는(환각) 걸 막고, **우리가 준 자료 안에서만** 판단하게 하려고. 우리 프로젝트에선 "학생이 올린 강의자료가 근거"라는 게 핵심 — 그래야 "네 1강 3페이지에 이거 나오는데 알아?" 식으로 정확하게 짚어줄 수 있음.

## 2. 우리 코드로 매핑 (`app.py` 열어서 같이 읽기, 20분)

| 파이프라인 단계 | 함수 | 뭘 하는지 |
|---|---|---|
| 청킹 | `chunk(text, size=500)` | 텍스트를 500자 단위로 자름 |
| 임베딩 | `embed(text)` | Gemini API로 텍스트 → 숫자 벡터 변환 |
| 저장(인덱싱) | `retrieve()`의 `_VECS` | 벡터를 numpy 배열에 그냥 쌓아둠 (벡터DB 없음) |
| 검색 | `cosine_topk(mat, q, k)` | 질문 벡터와 가장 가까운 청크 k개 찾기 (코사인 유사도) |
| 생성 | `answer(messages)` | 찾은 청크 + 규칙(`system_prompt.md`)을 Gemini에 넘겨서 답 생성 |

**직접 실행해서 확인:**
```bash
python app.py
# selfcheck OK - chunks=41  ← 청킹 결과 숫자
```

`cosine_topk`는 사실 3줄이야 (`app.py` 46~48줄). 벡터 두 개가 "얼마나 같은 방향을 보는지"를 계산하는 게 전부고, 이게 "의미가 비슷한 문장 찾기"의 정체임.

## 3. 각자 해볼 실습 (30분, 코드 몰라도 됨 — 숫자만 바꿔보기)

`.env`에 `GEMINI_API_KEY` 넣고 서버 띄운 상태에서 (`README.md` 참고):

1. **청크 사이즈 바꿔보기** — `app.py`의 `chunk(text, size=500)` → `size=200`으로 바꾸고 재시작. 검색 결과(답변 품질)가 어떻게 달라지나?
2. **top-k 바꿔보기** — `retrieve(query, k=4)` → `k=1` vs `k=8`. 근거를 너무 적게/많이 주면 어떻게 되는지 비교.
3. **근거 없는 질문 던지기** — 강의에 없는 걸 물어보기 (예: "고양이 키우는 법 알려줘"). RAG가 "모른다"고 하는지, 아니면 지어내는지 관찰.
4. **다른 강의 올려서 검색해보기** — 업로드 버튼으로 아무 텍스트나 넣고, 그 안의 특정 문장을 질문해서 실제로 찾아오는지 확인.

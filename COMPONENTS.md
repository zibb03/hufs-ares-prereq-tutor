# 학습 페이지(/chat) 컴포넌트 정리

> ARES · 선수개념 진단 RAG 학습 튜터
> 이 문서는 **실제 배포된 `/chat` 화면**에 들어간 컴포넌트를 하나씩 정리한다. 각 항목은 **넣은 내용 / 교육적 근거 / 이 위치에 둔 이유** 3단으로 서술한다.

## 설계 목표
"답을 바로 주는 챗봇"이 아니라, **학습자가 스스로 사고하고 오류를 교정하도록 돕는 선수개념 진단형 튜터**를 만드는 것. 핵심 원칙은 두 가지다.

1. **답이 아니라 사고를 스캐폴딩한다** — 정답은 최후에만, 그 전엔 질문·힌트로 스스로 도달하게.
2. **학습 도구를 별도 대시보드에 가두지 않는다** — 힌트·빈칸·근거가 **대화 흐름 속에서** 나타나 인지 흐름을 끊지 않게.

## 전체 정보 구조
```
┌──────────────┬─────────────────────────────────┐
│ 진행 현황     │        학습 컨텍스트 헤더        │
│ (좌 사이드)  ├─────────────────────────────────┤
│              │      소크라테스 대화 (중앙)      │
│  진단 맵     │   · 힌트/빈칸 카드 (인라인)      │
│  세션 목록   │   · 빠른 응답 선택칩             │
│              │   · 강의 근거칩                  │
│  끝내기 →    ├─────────────────────────────────┤
│  성찰        │  힌트·빈칸·자기설명 도구 + 입력창│
└──────────────┴─────────────────────────────────┘
```
- **좌측**: 전체 학습 경로와 현재 위치를 계속 보여주는 탐색 공간.
- **중앙**: 학생이 지금 답해야 할 질문에 집중하는 주 작업 공간. 힌트/빈칸/근거가 이 흐름 안에 등장.
- **하단**: 막혔을 때 손이 닿는 학습 도구 + 입력창.

우측 고정 패널을 **두지 않은 것**이 이 버전의 선택이다. 도구를 우측에 모으면 정보 밀도는 높지만 시선이 분산되고 인지부하가 커져(EDM 2024), 도구를 대화 흐름과 입력창 옆으로 옮겨 "필요한 순간, 보던 자리에서" 쓰게 했다.

## 1. 학습 컨텍스트 헤더

### 넣은 내용
- 현재 과목명 배지(선택한 과목에 따라 바뀜)
- 현재 학습 개념
- 학습 단계 배지(진단 → 선수개념 보충 → 적용 → 자기설명)
- 이번 학습 목표 한 줄

### 교육적 근거
학습자가 "지금 무엇을, 어느 단계에서 배우는가"를 잃으면, 내용 이해에 써야 할 작업기억을 화면 탐색에 낭비하게 된다. 현재 맥락과 목표를 지속적으로 노출하는 것은 방향 상실을 줄이고 인지부하를 낮춘다(ITS 설계 원칙).

### 이 위치에 둔 이유
헤더는 대화를 읽기 전에 가장 먼저 보이고 스크롤과 무관하게 같은 자리에 있다. 매 질문마다 맥락을 반복 설명하지 않아도 "지금 어디를 배우는 중인지"가 유지된다. 기존 `lecture-badge`를 확장해 강의자료 업로드 흐름도 그대로 보존했다.

## 2. 진행 현황 — 선수개념 진단 맵 + 진행률 (좌측)

### 넣은 내용
- 확률변수 → 결합분포 → 조건부확률 → 기댓값의 개념 순서
- 각 개념의 상태: 점검 전 / 학습 중 / 숙달 (색 + 텍스트 병기)
- 전체 진행률 막대(%)

### 교육적 근거
숙달학습에서는 다음 개념으로 넘어가기 전에 선수개념 결손을 확인·보충해야 한다. 또한 "무엇을 알고 무엇을 모르는지"를 외부화하면 학습자가 자기 이해를 점검하는 메타인지가 촉진되고, 작은 진행 변화는 학습 지속에 필요한 통제감을 준다.

### 이 위치에 둔 이유
진단 맵은 대화의 세부보다 상위 정보이므로 좌측 사이드바에 **얇게** 배치했다. 중앙에 두면 매 답변마다 시선을 방해하지만, 좌측 주변시야에 두면 필요할 때만 확인한다. 큰 대시보드로 키우지 않은 이유 = 진단 결과는 '확인용'이지 '작업용'이 아니기 때문. 상태는 색뿐 아니라 텍스트로도 표시해 **색상에만 의존하지 않도록**(접근성) 했다.

> ⚠️ **데모 한계**: 현재 숙달 상태·진행률은 대화 턴 수를 이용한 데모용 시각화다. 실제 학습 성취 판정이 아니다. 운영 단계에서는 모델의 구조화된 진단 결과나 별도 확인 문제의 채점 결과로 갱신해야 한다.

## 3. 소크라테스 대화 영역 (중앙, 핵심)

### 넣은 내용
- 답을 바로 제시하지 않고 짧은 질문으로 이어가는 중앙 대화
- 학생 답변을 가장 최근 강의 근거(RAG)와 함께 처리하는 흐름

### 교육적 근거
설명을 일방 전달하기보다 질문·반문으로 기존 지식을 꺼내고 오류를 스스로 교정하게 하는 방식이 능동적 사고를 유도한다. 간접 힌트와 짧은 질문은 학생을 문제 해결의 주체로 남게 한다(system_prompt.md의 "정답을 바로 말하지 않는다" 원칙과 일치).

### 이 위치에 둔 이유
대화가 학습의 핵심 활동이므로 화면에서 가장 넓은 중앙을 사용했다. 나머지 컴포넌트는 모두 이 대화를 돕는 보조이며, 학생의 답변과 튜터의 질문이 시각적 우선순위 1순위가 되게 했다.

## 4. 빠른 응답 선택칩 (대화 흐름 안)

### 넣은 내용
- 시작 개념을 고르는 선택지("확률변수부터 점검하고 싶어요" 등)
- "확인 문제 주세요", "더 작은 예가 필요해요", "아직 잘 모르겠어요" 같은 상황별 빠른 응답
- 선택지 대신 직접 입력할 수 있는 텍스트 영역(항상 열림)

### 교육적 근거
항상 자유 입력만 요구하면 학생은 학습 내용뿐 아니라 "어떻게 답을 표현할지"까지 동시에 결정해야 한다. 제한된 선택지는 표현·조작 부담을 줄여 개념 판단에 집중하게 한다(EDM 2024). 다만 매번 객관식만 쓰면 추측·재인에 의존할 수 있으므로 직접 입력을 항상 열어둔다.

### 이 위치에 둔 이유
선택지는 응답해야 할 튜터 메시지 **바로 아래** 대화 흐름에 배치했다. 질문과 선택지가 멀리 떨어지면 둘을 작업기억에서 연결하는 비용이 생기기 때문이다.

## 5. 학습 도구 3종 — 입력창 바로 위 (힌트 · 빈칸으로 연습 · 내 말로 설명)

세 도구를 **입력창 바로 위 동일 폭 버튼**으로 묶었다. 공통 배치 이유: **막혔을 때 손이 가장 먼저 닿는 곳이 입력창 옆**이기 때문. 답을 쓰려다 막히는 바로 그 지점에서, 시선을 딴 데로 옮기지 않고 개입할 수 있다.

### 5-1. 단계형 힌트 위계
- **넣은 내용**: ① 아는 값/구할 값 관찰 → ② 선수개념 연결 → ③ 작은 예제로 첫 단계 → (그래도 막히면) 튜터에게 최소 단서 요청. 누르면 힌트가 **대화 흐름 안에 카드로** 등장.
- **교육적 근거**: 정답을 바로 주면 자기교정 기회가 사라진다. 약한 단서에서 강한 단서로 점진 제공하는 것이 "생산적 어려움"을 유지하는 스캐폴딩이며, 정답 공개는 최후로 미룬다(arXiv 2602.19303).
- **왜 인라인**: 힌트가 별도 패널이 아니라 대화 안에서 나오면 "튜터가 지금 건네주는 것"이라는 맥락이 유지돼 사고 흐름이 끊기지 않는다.

### 5-2. 워크드 예제 → 페이딩 (빈칸으로 연습)
- **넣은 내용**: 동전 두 번 던지기 예제에서 확률변수의 가능한 값을 학생이 **빈칸으로 직접 채우게** 하는 카드(대화 흐름 안).
- **교육적 근거**: 초보자는 절차 자체가 익숙하지 않아 모든 단계를 동시에 처리하기 어렵다. 처음엔 완성된 예제로 절차를 보여주고 점차 지원을 줄이면(fading), 초기 인지부하를 낮추면서 독립적 문제 해결로 전환된다.
- **왜 여기**: 예제는 대화를 대체하는 본문이 아니라 필요할 때 참고하는 발판이므로, 힌트로도 연결이 안 될 때 더 구체적인 예제로 내려가도록 도움 강도 순서를 만들었다.

### 5-3. 자기설명 프롬프트 (내 말로 설명)
- **넣은 내용**: "왜 그런지 내 말로 설명하면…" 문장 틀을 입력창에 열어 서술형 응답을 유도.
- **교육적 근거**: 자기설명은 배운 내용을 그대로 반복하는 게 아니라 개념 간 이유·연결을 자기 언어로 재구성하게 한다. 이 과정에서 겉핥기 이해가 드러나고 장기기억으로의 정교화가 일어난다 — 학습효과가 가장 큰 활동(arXiv 2602.19303).
- **왜 여기**: 버튼을 누르면 중앙 입력창으로 문장이 들어와, 자기설명이 별도 메모로 끝나지 않고 튜터의 후속 질문으로 이어진다.

## 6. 강의 근거칩 (답변 아래 인라인)

### 넣은 내용
- RAG 검색에서 **실제 사용된 청크**의 출처를 "근거 · N쪽" 칩으로 답변 바로 밑에 표시(마우스오버 시 발췌 미리보기).

### 교육적 근거
생성된 답변이 어떤 강의 내용에 근거하는지 보여주면, 학생이 출처를 검토하고 원자료로 돌아갈 수 있다. 이는 모델 답변을 무비판 수용하는 대신 근거를 확인하는 비판적 읽기를 돕고, 근거 없는 생성(환각)에 대한 신뢰 위험을 낮춘다.

### 이 위치에 둔 이유
근거는 그 답변 **바로 옆**에 있어야 즉시 대조된다. 별도 근거 패널로 빼면 대조 비용이 커지므로, 답변과 직접 연결되는 페이지 칩을 메시지 아래에 짧게 두는 점진적 공개 방식을 택했다.

## 7. "끝내기 →" 버튼 → 세션 성찰 (좌측 하단)

### 넣은 내용
- 끝내기 버튼 → 성찰 모달(오늘 배운 것 / 아직 헷갈리는 것 / 현재 설명 가능 수준)
- 제출한 성찰은 대화로 보내져, 튜터가 다음 학습 한 단계만 제안하며 마무리

### 교육적 근거
자기조절학습은 계획·수행만으로 끝나지 않고 **결과를 평가해 다음 계획으로 연결**해야 사이클이 닫힌다(arXiv 2511.04144). 배운 것을 회상하고 남은 혼란을 명시하면 학습자는 이해를 점검하고 다음 행동을 구체화한다.

### 이 위치에 둔 이유
세션 종료는 '끝 지점'의 행동이므로 사이드바 맨 아래에 상시 접근 가능한 버튼으로 두었다. 명시적 끝내기 버튼으로 "그냥 창을 닫는" 대신 **성찰하고 닫는** 습관을 유도한다. 작성 시엔 모달로 집중 환경을 만들어 대화의 다른 정보와 경쟁하지 않게 했다.

## 8. 세션 목록 · 강의자료 업로드 (좌측 사이드)

### 넣은 내용
- 지난 학습 세션 목록(localStorage), 새 학습 세션
- 강의자료 업로드(.txt/.md/.pdf → 클라이언트 텍스트 추출 → RAG 재인덱싱)

### 교육적 근거
학습 세션은 누적되어야 한다. 이전 대화를 다시 열면 자신의 이해 변화와 결손 보강 과정을 확인할 수 있다. 자료 업로드는 "강의 근거 기반 튜터"라는 전제를 유지한다.

### 이 위치에 둔 이유
세션 관리·업로드는 학습 내용 자체보다 상위 작업이므로 본문 학습 루프 밖(사이드바)에 두었다.

---

## 전체 학습 루프
컴포넌트는 다음 순서로 연결된다.
1. **헤더 + 진단 맵**으로 목표와 현재 위치를 확인한다. (계획)
2. **중앙 질문 + 선택칩**으로 선수개념을 진단한다. (진단·모니터링)
3. 막히면 **힌트 · 빈칸 예제**를 필요한 만큼 대화 안에서 사용한다. (스캐폴딩)
4. 이해한 내용을 **내 말로 설명**한다. (자기설명)
5. **강의 근거칩**으로 답의 출처를 확인한다. (비판적 읽기)
6. **끝내기 → 성찰**로 배운 것·남은 혼란을 평가하고 다음을 계획한다. (평가)

목적은 기능을 많이 보여주는 게 아니라, **계획 → 진단·모니터링 → 스캐폴딩 → 자기설명 → 평가**가 한 화면 안에서 끊기지 않게 만드는 것이다.
